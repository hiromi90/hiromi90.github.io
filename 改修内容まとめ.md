## 2. 設計方針（重要な決定事項）

- **編集用データは .js ではなく .txt に統一**した。
  理由：`.js` はカンマ・引用符の1文字ミスでサイト全体が壊れるため、
  プログラミング未経験者には不向き。メモ帳で編集できる `.txt` に変更。
- サイトは起動時に `content/*.txt` を `fetch` で読み込み、JavaScriptで描画する。
  → **ローカルではLive Server等、公開はNetlify**（従来と同じ運用）。
  HTMLの直接ダブルクリックでは表示されない旨をREADME冒頭に明記。
- SNSリンクは各フッターのHTMLを編集せず、`content/site.txt` の値で
  **href とアカウント名テキストを実行時に自動差し替え**する方式にした。

## 4. GAS（Apps Script）運用メモ

- **personal.gs**：スプレッドシートIDで直接開き、全体を try/catch で保護。
  シートが無い/空でも `[]` を返し、例外は `{error:…}` をJSONで返す（500を出さない）。
  差し替え時は **必ず「新しいバージョン」でデプロイ**し、アクセスを「全員」にする。
- **results.gs**：年度スプレッドシートの全シート（タブ）を
  `[{ sheet:"タブ名", data:[行オブジェクト...] }]` の形で返す。
  「_」始まりのタブは除外。年度ごとにデプロイし、URLを `results-config.js` に登録。

| 安全7 | 公開/安全 | 公開物から除外する開発用ファイルを定義（`.git/`・`.gitignore`・`.DS_Store`・`preview-*.html`・`publish/`・`*.zip`）。Netlify 公開は **公開用コピーフォルダ方式** に統一し、rsync／robocopy のコマンド例を README に明記 | ✅ |

### 設計方針・決定事項

- **危険度は `.git/` ≫ `.gitignore`**：`.git/` は全履歴＋削除済みファイルまで復元可能で**致命的**。
  一方 `.gitignore` は除外リストのテキストに過ぎず、**中身のデータは漏れない**。
  それでも公開しない理由は次の3点：
  1. 内部のファイル名を開示し「そこに何かある」というヒントを与える
  2. Git 管理下であることを示し、`/.git/` を探す自動スキャンを誘発する
  3. サイトの表示に一切使われない（＝配信する理由がない）
- **`.gitignore` はドロップ公開に効かない**：`.gitignore` は Git のみのルール。
  フォルダをブラウザにドロップする公開方法では Git が動かないため無効。
  除外の実体は**公開用コピーフォルダ**が担う（`--delete` / `/MIR` を必須とし、
  整理済みの旧記事が公開先に残留しないようにする）。
- **記載方針（厳守）**：`.gitignore` に**中身を推測させる名前を書かない**
  （例：`名簿.xlsx`・`部員連絡先`）。GitHub Pages では `.gitignore` はコミット必須で
  「上げない」選択が取れず、配信可否は Jekyll 処理の有無（`.nojekyll` の設置状況）に
  依存して確実でないため、**記載内容を無害な汎用名に保つことが唯一確実な対策**。
  非公開にしたいファイルは `.gitignore` に列挙せず、サイトフォルダの外へ置く。
- **`news/_drafts/` は除外しない（今回の決定）**：下書きも履歴に残して復元可能にするため。
  サイト上の非表示は従来どおり `_drafts/` 運用で担保される。
- **ホスティング別の整理**：`.git/` 露出は **Netlify のドロップ公開に固有**のリスク
  （GitHub Pages は成果物のみ配信するため `.git/` は配信対象外）。
- **既存コミットには遡及しない**：`.gitignore` は「これから追加する物」のみ対象。
  既に追跡されているファイルの除外には `git rm --cached` が別途必要。

### netlify.toml の不具合（修正9）の詳細

- **症状**：`from = "/*/.DS_Store"` が何にもマッチせず、**階層下の `.DS_Store` が素通り**していた
  （`/images/.DS_Store`・`/content/.DS_Store`・`/images/members/.DS_Store` 等）。
  ルート直下の `/.DS_Store` はワイルドカード不使用のため正常に塞げていた。
- **原因**：Netlify の仕様上、**アスタリスクはパスの末尾でしか使えない**
  （公式ドキュメント：`/jobs/*.html` のような途中のワイルドカードは不可、`/jobs/*` は可）。
- **なぜ最後の `/*` → 404 で救われないか**：あのルールには `force` が無い。
  **force なしのルールは「実ファイルが存在しない場合のみ」適用される**ため、
  `.DS_Store` が実在すればそのまま配信される。`.DS_Store` 系のルールにだけ
  `force = true` が付いているのはこのため。
- **対応**：プレースホルダ（`:名前`＝1階層分にマッチ）方式へ変更。
  本サイトの最大の深さは2階層（`/images/members/`・`/news/フォルダ名/`）のため2本で網羅。
  **階層が増えた場合はルールを1本追加すること**。
- **`/*` → 404 の最終ルールは残置**（2026-07-15 判断）。Netlify は 404.html を
  自動で返すため実質は冗長だが、意図の明示のため残す。**`force` を付けないこと**
  （付けると全ページが404になる）。
- **netlify.toml の仕様メモをファイル冒頭コメントに追記**（* は末尾のみ／force の意味／
  ルールは上から順に評価され最初の一致が使われる）。

### 実装の要点

- **HTML構造**：大会1件ごとに `.comp-item` で包み、その中を「`.comp-head`（見出し・ボタン）＋ `.comp-body`（記録テーブル）」の2段構成にした。
  ```
  .comp-item
    ├ .comp-head[role=button][aria-expanded][aria-controls]
    │   ├ .comp-name / .comp-meta
    │   └ .comp-toggle  … 下中央の印（CSSの枠線で描画。画像は使わない）
    └ .comp-body#comp-body-{年度}-{連番}[hidden]
        ├ .scroll-hint
        └ .table-wrap > table.records-table
  ```
- **開閉の仕組み**：`hidden` 属性の付け外しで切り替える。`aria-expanded` の値を印の向き（CSS）と連動させているため、JS側で印を直接いじる必要がない。

- **データ未入力の大会**：「この大会のデータはまだ入力されていません。」の案内文も `.comp-body` の中に入れたため、他の大会と同じように開閉できる。

## 15. 2026-07-23 改修（第14回・ホバー時の不具合修正）

| No. | 種別 | 内容 | 対応 |
|-----|------|------|------|
| 修正1 | 大会結果（/records/results） | 見出しボックスに**カーソルを合わせても薄くならない**不具合を修正。 | ✅ |

### 原因（CSSの詳細度の競合）

第13回で見出しボックスに `fade-up`（スクロール連動フェード）を付けたままにしていたことが原因。

| 記述位置 | ルール | 詳細度 |
|---|---|---|
| `css/style.css` 700行目付近 | `.comp-head:hover { opacity: 0.6; }` | 0-2-0 |
| `css/style.css` 850行目付近 | `.fade-up.visible { opacity: 1; }` | 0-2-0 |

詳細度が**同点**のため、CSSの規則により**後に書かれている方（`.fade-up.visible`）が優先**され、
`opacity: 1` が確定してホバーの `opacity: 0.6` が打ち消されていた。
加えて `.fade-up` の `transition` は 0.65 秒で、仮に効いても反応が鈍く感じられる状態だった。

### 対処

`fade-up` を **`.comp-head`（見出しボックス）から外側の `.comp-item` へ移動**した。
これでフェード用の `opacity` とホバー用の `opacity` が別々の要素に分かれ、競合が根本的に解消する。

- `!important` は使わない（他の箇所に波及し、将来の調整が困難になるため）。
- ホバー規則は念のため `.comp-item .comp-head:hover`（0-3-0）に変更し、詳細度を1段上げて再発を防ぐ。
- フェードの見え方は変わらない（大会1件のまとまりごとに、従来どおり 0.65 秒でふわっと表示される）。
- ホバーの反応速度は `.comp-head` 本来の `transition: opacity 0.2s` に戻る。

### 変更ファイル

| ファイル | 変更内容 |
|---|---|
| `js/main.js` | `class="comp-head fade-up"` → `class="comp-head"`／`class="comp-item"` → `class="comp-item fade-up"`（2か所）。理由をコメントで明記 |
| `css/style.css` | ホバー規則を `.comp-item .comp-head:hover` に変更。`.comp-head` に `fade-up` を付けてはいけない旨をコメントで明記 |

### 検証（headless Chromium ＋ jsdom）

**実ブラウザ（Chromium）による確認・16項目すべて合格**

- 構造：見出しに `fade-up` が無い／`.comp-item` に `fade-up` がある／スクロール後に `visible` が付く
- ホバー前：まとまり・見出しとも `opacity: 1`
- ホバー中：見出しが `opacity: 0.6` になる／まとまり側は変化しない／`cursor: pointer`
- ホバー解除：`opacity: 1` に戻る
- 両立：ホバーしたまま押しても開く／開いた状態でもホバーで薄いまま／印が反転している
- 別の大会へ移動：2件目が薄くなり、1件目は元に戻る
- スマホ相当（`has_touch` / `is_mobile`）：**薄くならない**（誤作動防止が意図どおり効いている）／タップで開く

**回帰テスト（jsdom・第13回の18項目）も全項目合格**（開閉・複数同時展開・キーボード操作・年度切替に影響なし）。

### 学び（今後の実装で注意する点）

- **`fade-up` と、ホバー等で `opacity` を変える処理は同じ要素に付けない。** 付ける場合は入れ子の別要素に分ける。
- 詳細度が同点のときは記述順で決まる。`style.css` の後半（800行目以降）に汎用クラスが並んでいるため、
  前半に書いた個別スタイルが打ち消されやすい。`opacity` / `transform` / `display` を扱う際は特に確認する。
- CSSの競合は jsdom では検出しにくい。**ホバーやアニメーションが絡む変更は headless Chromium で計算値を確認する**。
